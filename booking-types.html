Booking types
Four types. Different dispatch logic for each.
These are not just UI variants. Each type has different feasibility rules, cost scoring weights, re-optimisation behaviour, and failure paths. Click any type to expand its dispatch logic.

Hourly — mid-trip driver handover
When the block doesn't fit one shift
When an hourly booking is assigned, the system checks whether the full block fits within the driver's remaining shift including overtime allowance. If not, it evaluates two options and picks the cheaper one.
Decision logic
Runs at assignment time. If block fits → no handover. If not → compare Option A vs Option B.
Option A — Keep original driver
Cost A =
  overtime_rate_per_min
  × minutes_block_extends_beyond_shift_end
Always available as fallback — no dependency on second driver
Chosen automatically when no second driver is found
Chosen if Option B cost ≥ Option A cost
Option B — Handover to fresh driver
Cost B =
  cost_per_empty_km
  × empty_km (incoming driver → handover point)
+ any overtime for outgoing to reach safe stop
Only considered if a suitable second driver is available
Chosen only if Cost B < Cost A
How the handover executes
1
System selects a handover point — a natural pause in the rider's itinerary or nearest safe stop to the shift boundary.
2
Outgoing driver always completes their current leg before stopping. Handover never happens mid-drive.
3
Incoming driver notified in advance with handover location and time.
4
Rider notified that a driver change will occur and at what point. (Consent vs notification — open question #9.)
5
At handover point: outgoing driver checks out, incoming driver checks in. Block continues uninterrupted.
City-to-city
Driver and vehicle bound for the full duty
After a city-to-city dropoff, the vehicle must return to its home base before the duty closes — via a revenue return leg or a deadhead. The return deadhead cost is pre-priced into the base fare.

Post-dropoff behaviour

Wait window starts at dropoff (duration configurable per route)
If vehicle type is operable in destination → enters local trip pool with home base return credit applied
If vehicle type not operable → sits idle, available for return legs only
Re-optimisation continuously searches for return legs during wait window
When wait window expires → final re-opt attempt, then overnight vs deadhead decision

Additional feasibility check (city-to-city only)

Shift end + max overtime
≥ est. dropoff
  + city wait window
  + travel time (destination → home base)
Driver and vehicle are bound together for the full duty. City-to-city trips cannot be split between duties.
Overnight vs deadhead decision

Option A — Deadhead home now

=cost_per_empty_km × km_to_home_base
+expected_future_deployment_cost
Expected future deployment cost = probability of confirmed pickup in destination within lookahead window × cost_per_empty_km × km from home to destination

Option B — Overnight stay

=overnight_compensation_cost
+cost_per_empty_km × km_to_home_base
(deferred to next shift)
Lookahead window: 24 hours (configurable). Engine re-evaluates at next shift start.
Decision: pick whichever option has the lower total cost. Engine runs this comparison once at wait window expiry.

Deployment trips
Last resort — always
A deployment trip positions a vehicle from its home base to a city where no vehicle is available. Used only when no local or in-transit option exists. Subject to both a feasibility check and a revenue justification check.
Priority order — always exhausted before deployment
1Vehicle already at destination
2Vehicle en route to destination
3Deployment from home base ← last resort

Feasibility check

Travel time:
home base → pickup city → trip dropoff → home base
≤ remaining shift time + max overtime allowance
Full round trip must fit within the duty's remaining time window including overtime.

Revenue justification check

Trip revenue
≥ min_deployment_revenue_ratio
  × cost_per_empty_km
  × empty_km (home base → pickup city)
Only the outbound positioning cost is checked. Return deadhead is already priced into the base fare.
